Overview

snap-dashboard is a self-hosted web application for managing Ubuntu snap packages. It keeps track of your published snaps, monitors channel versions across architectures, surfaces open GitHub issues and pull requests, and integrates with YARF (Yet Another Robot Framework) to run automated tests against snaps before promoting them to stable.

📦
Channel tracking

See stable, candidate, beta, and edge versions per architecture side-by-side.

🐛
Issue & PR tracking

Open GitHub issues and PRs from packaging and upstream repositories, in one place.

🧪
Automated YARF testing

Trigger GitHub Actions test runs against any snap, see results and screenshots inline.

🚀
One-click promotion

Promote a tested revision to stable directly from the dashboard.

First login

snap-dashboard uses GitHub OAuth for authentication. Click Sign in with GitHub on the login page and authorise the app.

First-user admin

The first GitHub account to log in becomes the administrator automatically. As admin you can:

  • Add other users to the access allowlist via the Admin page
  • Toggle admin status for other users

All subsequent logins require the account to be on the allowlist first.

Onboarding

After your first login, an onboarding wizard guides you through setup:

  1. Publisher name — your Snap Store publisher account name (e.g. ken-vandine). snap-dashboard uses this to auto-discover all your published snaps.
  2. GitHub token — optional, but recommended. A Personal Access Token increases the GitHub API rate limit from 60 to 5,000 requests/hour, enabling issue and PR tracking and test workflow dispatch.
  3. First collection — once you click "Go to Dashboard" the first collection run is triggered in the background. It may take a minute or two to fetch all snap data.

You can change any of these settings later from Settings.

Dashboard overview

The dashboard shows all your tracked snaps at a glance. The Attention Needed section at the top highlights snaps where a newer version exists in candidate/edge than stable — these are candidates for testing and promotion.

Channel version badges

  • green — same as stable (or is stable)
  • amber — different from stable (newer)
  • — — not published in this channel
  • — — not published in stable at all

Refresh All

Clicking Refresh All triggers a background collection run that fetches updated channel data and GitHub issues for all tracked snaps. You can also refresh a single snap from its detail page.

Adding snaps

Snaps are added in two ways:

Auto-discovery

When a publisher is configured, the first collection run fetches all snaps published by that account from the Snap Store. This happens automatically.

Manual addition

Use Add Snap to add any snap by name — even ones you don't publish. Search by snap name, then optionally set the packaging repo and upstream repo URLs for GitHub issue tracking.

Repository URLs

For each snap you can set two repo URLs:

  • Packaging repo — where the snapcraft recipe lives (e.g. a snap/ folder or dedicated snap packaging repo)
  • Upstream repo — the upstream project repo (may be the same as packaging, or separate)

For auto-discovered snaps, these are populated automatically from Snap Store metadata when available.

Channel map

The snap detail page shows the full channel map — every published channel/architecture combination with its version and revision number. Versions that differ from stable are highlighted amber.

Channels are collected on a schedule (configurable in Settings, default: every 6 hours) or on demand via the Refresh button.

Issues & Pull Requests

For each snap with repository URLs set, snap-dashboard fetches all open GitHub issues and PRs. These are shown on the snap detail page and counted in the dashboard table.

The snap detail page has filter tabs to view:

  • All issues and PRs together
  • Issues only / PRs only
  • Packaging repo items / Upstream repo items (if both repos are set)
Rate limits: Without a GitHub token, GitHub API calls are limited to 60/hour. With a token, the limit is 5,000/hour. Set your token in Settings.

YARF testing — overview

Testing integrates with YARF (Yet Another Robot Framework) to run automated tests against snaps in CI, then report results back to the dashboard.

How it works

  1. snap-dashboard finds snaps with a newer version in candidate or edge than stable and lists them on the Testing page.
  2. You click Run Tests (or enable auto-test). snap-dashboard dispatches a workflow_dispatch event to your testing repository on GitHub.
  3. The GitHub Actions workflow installs YARF, the snap, and runs your test suite. Screenshots and results are committed to a result branch.
  4. A Pull Request is opened in the testing repo summarising the results.
  5. snap-dashboard polls GitHub and updates the run status live (pending → running → passed/failed).
  6. For passed candidate runs, a Promote to Stable button appears.

Candidate vs Edge

  • Candidate — has a promotion path. After tests pass you can promote directly to stable from the dashboard.
  • Edge — test-only. No direct promotion path; edge snaps need to be moved to candidate first via snapcraft release before promotion.

Testing repository setup

You need a dedicated GitHub repository to hold your YARF suites and the GitHub Actions workflow.

Step 1 — Create the repository

Create a new GitHub repository (e.g. you/snap-tests). It can be private.

Step 2 — Add the workflow

Download the workflow template from here and commit it to .github/workflows/snap-test.yml in your testing repository.

Step 3 — Add a repository secret

In your testing repository: Settings → Secrets and variables → Actions → New repository secret

  • Name: SNAP_DASHBOARD_GITHUB_TOKEN
  • Value: a GitHub Personal Access Token with repo scope

This token lets the workflow push result branches and open PRs.

Step 4 — Configure snap-dashboard

In Settings, set Testing Repository to owner/repo (e.g. kenvandine/snap-tests) and save.

YARF suite structure

Test suites live under suites/ in your testing repository:

suites/
└── mysnap/
    └── suite/
        ├── __init__.robot      ← Suite setup/teardown
        └── test_mysnap.robot   ← Test cases

The workflow looks for suites/<snap_name>/suite/__init__.robot to confirm a suite exists before offering the Run Tests button.

__init__.robot example

*** Settings ***
Library    Collections
Library    Process
Library    OperatingSystem

Suite Setup     Suite Setup Steps
Suite Teardown  Suite Teardown Steps

*** Keywords ***
Suite Setup Steps
    # DISPLAY=:99 is set by the CI workflow via Xvfb
    Log    Suite starting on display $ENV{DISPLAY}

Suite Teardown Steps
    Log    Suite complete

Test file example

*** Settings ***
Library    OperatingSystem
Library    Process

*** Test Cases ***
Snap Is Installed
    ${rc}=    Run And Return RC    snap list mysnap
    Should Be Equal As Integers    ${rc}    0

Application Launches
    ${proc}=    Start Process    mysnap    alias=app
    Sleep    3s
    Process Should Be Running    app
    Terminate Process    app

Screenshots

Any PNG files written to results/<snap_name>/ during the test run are committed to the result branch and embedded in the test PR body. Use Robot Framework screenshot keywords or any tool that writes PNGs to that directory.

*** Settings ***
Library    Screenshot

*** Test Cases ***
Capture App Screenshot
    Take Screenshot    results/mysnap/main_window.png

Running tests

From the dashboard

The Testing page lists all snaps with a newer version in candidate or edge than stable. Click Run Tests next to any snap with a suite configured.

The run status updates live (polling every 15 seconds): pending → running → passed or failed.

Click the ↗ link next to a running badge to open the GitHub Actions log directly.

Stuck runs

If a run gets stuck in pending or running state (e.g. the workflow was cancelled externally), click the ✕ button in the Triggered By column to mark it failed and unblock the UI.

Sync from GitHub

Click Sync from GitHub at the top of the Testing page to re-read all open test PRs from GitHub and reconcile their status with the local database. Useful after manual changes or if the auto-poll missed an update.

Auto-test

Enable Enable automatic testing in Settings to automatically trigger a test run whenever a newer version is detected in candidate or edge during a collection. Requires a testing repository to be configured and a suite to exist for the snap.

Reading test results

Each test run opens a PR in your testing repository. Click the PR number in the Testing page to view the full result detail page, which shows:

  • A results table (snap, version, channel, architecture, status)
  • Screenshots (if any PNGs were written to results/<snap>/)
  • The list of files changed (committed results)
  • GitHub Actions run link for full logs
  • The Promote button (for passed candidate runs)

Promoting to stable

When a candidate snap passes tests, it appears in the Pending Promotion section at the top of the Testing page and on the PR detail page with a Promote to Stable button.

Clicking Promote to Stable:

  1. Runs snapcraft release <snap> <revision> stable on the server
  2. Marks the test run as promoted in the database
  3. Posts a comment on the test PR and closes it
  4. Redirects back to the Testing page
Edge snaps cannot be promoted directly to stable from the dashboard. Edge → Candidate → Stable is the expected flow. Move the snap to candidate first using snapcraft release <snap> <revision> candidate.

Settings reference

All per-user settings are saved in the database and can be changed at any time from Settings.

SettingDescription
Snap Store Publisher Your Snap Store account name. Used to auto-discover snaps on collection runs.
Collection Interval How often the background scheduler runs a full collection (1 / 6 / 12 / 24 hours).
GitHub Token Personal Access Token for GitHub API calls. Increases rate limit from 60 to 5,000 req/hour. Needed for issue/PR tracking and for triggering test workflows.
Testing Repository GitHub repo containing YARF suites and the workflow, in owner/repo format.
Enable automatic testing Automatically trigger tests when a new version is detected during collection.

GitHub token

A GitHub Personal Access Token (PAT) is required for:

  • Fetching open issues and PRs (rate-limited at 60/hr without a token)
  • Triggering YARF test workflows (workflow_dispatch)
  • Polling workflow run status and syncing test PR data

Creating a token

  1. Go to GitHub → Settings → Developer settings → Personal access tokens → Fine-grained tokens (or Classic tokens)
  2. For a Classic token, select scope: repo
  3. Copy the token and paste it into Settings → GitHub Token
This token is stored per-user in the database. It is only used server-side to make GitHub API calls on your behalf.

Admin & allowlist

The Admin page is visible to admins only. It has two sections:

Registered users

Shows everyone who has logged in, with their join date and last login. Admins can toggle other users' admin status (but cannot self-demote).

Access allowlist

Only GitHub accounts on the allowlist can log in. To grant access to someone else:

  1. Go to Admin
  2. Enter their GitHub username in the Add to Allowlist form
  3. Optionally add a note
  4. Click Add to Allowlist

They can now log in with their GitHub account. To revoke access, click Remove next to their entry.

Troubleshooting

Test runs stay in "pending" forever

The background poller waits up to 3 minutes to find the GitHub Actions run ID. If the dispatch failed (e.g. invalid token, workflow not found), the run stays pending.

  • Check that your GitHub token has repo scope and has not expired
  • Check that .github/workflows/snap-test.yml exists in your testing repository and has a workflow_dispatch trigger
  • Click the ↗ link next to the pending badge to check if a run appeared in GitHub Actions
  • Use the ✕ button to manually mark the run failed and try again

Tests pass but no screenshots appear

Screenshots are only included in the PR if PNG files are written to results/<snap_name>/ during the test run. Ensure your YARF test suite saves screenshots there.

Also check the GitHub Actions log for YARF errors — if YARF failed to start the Mir compositor (e.g. no display), no tests will run and no files will be created.

YARF fails with "Unable to connect to display"

YARF's Mir platform needs an X11 backend. The workflow installs xvfb and starts Xvfb :99 before running YARF. If you see this error on an older workflow version, download the latest workflow template from here and update your testing repository.

GitHub API rate limit errors

Set a GitHub Personal Access Token in Settings to increase the rate limit from 60 to 5,000 requests/hour.

Workflow dispatch returns 422

This means the workflow_dispatch trigger was not found. Common causes:

  • The workflow file has not been pushed to the main branch
  • The workflow file has YAML syntax errors (check GitHub → Actions → Workflows)
  • The workflow file is in a subdirectory other than .github/workflows/

Running YARF tests locally

You can run tests on your own machine without triggering a GitHub Actions workflow:

# Install YARF and dependencies
sudo snap install --beta yarf
sudo snap install mir-test-tools
sudo snap connect yarf:process-control

# Install the snap under test
sudo snap install mysnap --channel candidate

# Start a virtual display (Linux only)
sudo apt-get install -y xvfb
Xvfb :99 -screen 0 1920x1080x24 &

# Run the suite
mkdir -p results/mysnap
DISPLAY=:99 yarf \
  --platform Mir \
  --outdir results/mysnap \
  suites/mysnap/suite

# Check results
ls results/mysnap/
